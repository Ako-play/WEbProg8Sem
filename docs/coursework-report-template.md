# Шаблон отчёта по курсовой работе

## Тема

Разработка веб-приложения для конвертации криптовалют и просмотра истории котировок.

## Цель работы

Спроектировать и реализовать клиент-серверное веб-приложение, которое позволяет:

- регистрировать и аутентифицировать пользователей;
- получать актуальные котировки криптовалют через REST API;
- конвертировать криптовалюты;
- сохранять и просматривать историю курсов;
- разворачивать систему в контейнерах.

## Задачи

1. Проанализировать требования к системе.
2. Выбрать архитектуру frontend и backend.
3. Реализовать REST API на Go.
4. Реализовать авторизацию через JWT access + refresh token.
5. Спроектировать схему базы данных и подготовить миграции.
6. Реализовать React-клиент.
7. Подготовить Docker Compose и конфигурацию через переменные окружения.
8. Провести тестирование основных сценариев.

## Обоснование архитектуры

### Backend

В качестве backend выбран Go, так как он подходит для разработки высокопроизводительных REST-сервисов, имеет простую модель конкурентности и удобен для контейнеризации.

### Frontend

В качестве frontend выбран React, так как библиотека позволяет быстро реализовать интерфейс с переиспользуемыми компонентами и разделением по состояниям приложения.

### Авторизация

Используется схема JWT access + refresh token:

- access token передаётся в заголовке `Authorization: Bearer ...`;
- refresh token хранится в базе в виде хеша;
- при истечении access token выполняется обновление пары токенов.

### IoC и DI

Инверсия зависимостей реализована без контейнера:

- репозитории передаются в сервисы через конструкторы;
- сервисы передаются в HTTP-слой;
- композиция всего приложения сосредоточена в [backend/internal/app/app.go](backend/internal/app/app.go).

## Основные REST-методы

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/auth/me`
- `GET /api/v1/rates/latest`
- `GET /api/v1/rates/convert`
- `GET /api/v1/rates/history`

## Пользовательские сценарии

1. Пользователь регистрируется в системе.
2. Пользователь входит в систему.
3. Пользователь выбирает базовый и целевой криптоактив.
4. Система получает актуальную котировку и показывает результат конвертации.
5. Пользователь задаёт диапазон дат и получает историю курса.
6. Система сохраняет полученные данные в базе.

## Инфраструктура

Минимальная инфраструктура включает:

- `.env` для конфигурации;
- Docker Compose для запуска PostgreSQL, backend и frontend;
- SQL-миграции для подготовки схемы базы данных.

## Заключение

В результате была разработана веб-система для конвертации криптовалют и просмотра истории котировок, удовлетворяющая обязательным требованиям курсовой работы: REST API, авторизация, DI, база данных с миграциями и минимальная инфраструктура для развёртывания.
